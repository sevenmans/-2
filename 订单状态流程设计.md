# 体育馆预约系统 - 订单状态流程设计

## 1. 需求背景

当前体育馆预约系统需要完善订单状态流转逻辑，实现从创建到完成的全流程自动化管理，包括普通订单和拼场订单两种类型。系统需要清晰定义各状态间的转换条件、触发机制和业务规则。

## 2. 订单状态定义

### 2.1 基础状态（所有订单通用）

| 状态代码 | 状态名称 | 状态描述 |
|---------|---------|---------|
| PENDING | 待支付 | 订单已创建但尚未支付 |
| PAID | 已支付 | 订单已完成支付（预订成功） |
| VERIFIED | 已核销 | 用户已到场使用并完成核销 |
| COMPLETED | 已完成 | 订单全部流程结束 |
| CANCELLED | 已取消 | 订单被用户或管理员取消 |
| EXPIRED | 已过期 | 订单因超时未完成而自动过期 |

### 2.2 拼场订单特有状态

| 状态代码 | 状态名称 | 状态描述 |
|---------|---------|---------|
| SHARING | 拼场中 | 正在寻找拼场伙伴（对应后端 OPEN 状态） |
| SHARING_SUCCESS | 拼场成功 | 达到2人满员（发起者+加入者），等同于 PAID |
| OPEN | 开放中 | 拼场订单正在招募参与者 |
| APPROVED_PENDING_PAYMENT | 等待对方支付 | 发起者批准申请，等待申请者支付 |

## 3. 状态流转图

### 3.1 普通订单（独享场馆）状态流程

```
[创建订单] → [待支付] → [已支付] → [已核销] → [已完成]
                ↓           ↓          
                └─────→ [已取消] ←─────┘
                            ↑
                            │
                        [已过期]
```

### 3.2 拼场订单状态流程

```
[创建订单] → [待支付] → [已支付] → [拼场中] → [拼场成功] → [已核销] → [已完成]
                ↓          ↓          ↓           ↓           
                └────────────────→ [已取消] ←─────────────────┘
                                      ↑
                                      │
                                  [已过期]
```

## 4. 状态转换规则

### 4.1 手动触发转换

| 当前状态 | 目标状态 | 触发条件 | 操作权限 |
|---------|---------|---------|---------|
| PENDING | PAID | 用户完成支付 | 用户 |
| PENDING | CANCELLED | 用户取消订单 | 用户 |
| PAID | VERIFIED | 管理员核销订单 | 管理员 |
| PAID | CANCELLED | 用户取消订单 | 用户 |
| VERIFIED | COMPLETED | 管理员完成订单 | 管理员 |
| SHARING_SUCCESS | VERIFIED | 管理员核销订单 | 管理员 |
| SHARING_SUCCESS | CANCELLED | 用户取消订单 | 用户 |
| OPEN | CANCELLED | 用户取消拼场 | 用户 |

### 4.2 自动触发转换

| 当前状态 | 目标状态 | 触发条件 | 执行机制 |
|---------|---------|---------|---------|
| PENDING | EXPIRED | 创建后24小时内未支付 | 定时任务，每小时检查 |
| OPEN | CANCELLED | 开场前2小时未拼满 | 定时任务，每10分钟检查 |
| APPROVED_PENDING_PAYMENT | OPEN | 申请者支付超时 | 延时任务/定时检查 |
| PAID / SHARING_SUCCESS | EXPIRED | 预约时间过期未使用 | 定时任务，每天检查 |

## 5. 业务规则

### 5.1 订单取消规则

1. **待支付状态**：可随时取消，无限制
2. **已支付状态**：
   - 普通订单：预约时间前24小时可取消，退款100%
   - 拼场订单：预约时间前48小时可取消，退款100%
3. **拼场成功状态**：
   - 拼场订单：预约时间前24小时可取消，退款80%
4. **已核销状态**：不可取消
5. **已完成状态**：不可取消
6. **已过期状态**：不可取消

### 5.2 拼场规则

1. **拼场人数**：每个拼场订单固定为2人（发起者 + 1个加入者）
2. **拼场对象**：专为两支球队设计，每队1人代表
3. **拼场成功条件**：**当有1个用户申请加入并被批准时，达到2人满员，即视为拼场成功**
4. **拼场时限**：开场前2小时如果无人申请加入将自动取消
5. **拼场确认**：
   - 自动确认：有人加入并被批准即自动转为拼场成功状态
   - 手动确认：创建者需要批准申请者的加入请求
6. **拼场费用**：总费用由2人平均分配

## 6. 实现要点

### 6.1 数据库设计

1. **订单表扩展**：
   - 添加`status`字段，枚举类型，包含所有可能状态
   - 添加`bookingType`字段，区分普通订单和拼场订单
   - 添加`maxParticipants`字段，表示拼场最大人数
   - 添加`currentParticipants`字段，表示当前参与人数

2. **状态变更记录表**：
   - 记录每次状态变更的时间、原状态、新状态、触发方式（自动/手动）
   - 关联订单ID和操作用户ID

### 6.2 定时任务设计

1. **支付超时检查**：每小时执行一次，检查并处理超时未支付订单
2. **拼场超时检查**：每10分钟执行一次，检查并处理即将开场但未拼满的订单
3. **预约过期检查**：每天凌晨执行一次，检查并处理已过期未使用的订单

### 6.3 前端展示

1. **状态标签**：不同状态使用不同颜色标签直观展示
2. **操作按钮**：根据当前状态动态显示可执行的操作按钮
3. **倒计时显示**：
   - 支付倒计时：显示剩余支付时间
   - 拼场倒计时：显示距离自动取消还有多长时间
   - 取消倒计时：显示最后可取消时间

## 8. 预期效果

1. **用户体验提升**：状态流转清晰，自动化程度高，无需等待人工确认
2. **运营效率提高**：减少人工干预，自动处理常规状态变更
3. **数据一致性**：严格的状态转换规则确保数据一致性
4. **业务灵活性**：支持未来业务规则调整和扩展
