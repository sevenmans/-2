# 用户端核销功能优化方案

## 1. 核心确认
**你的观点完全正确。**
在 O2O 闭环中，“核销”是指**用户出示凭证**，**管理员进行验证**的过程，而非用户自己点击“签到”完成。
此前的“签到”按钮在逻辑上存在误导，必须修正为**“核销码”**或**“去核销”**，点击后展示核销凭证供管理员扫描或输入。

## 2. 核销码来源方案 (MVP P0级)
考虑到“最小闭环”原则，我们不引入复杂的动态码生成机制，而是直接利用现有的**唯一订单号**作为核销凭证。

### 方案：订单号 + 简码
- **数据源**：直接使用订单的 `orderNo`（订单号）。
- **展示形式**：
  1.  **完整订单号**：用于精确核销。
  2.  **核销简码**（可选）：取订单号后 6 位或 8 位，大号字体展示，方便管理员快速手动输入。
  3.  **二维码**（可选，视插件支持情况）：将订单号编码为二维码。鉴于 MVP 要求，若无现成组件，优先展示**大号数字码**。

## 3. 修改范围与计划

### 3.1 列表页 (`src/pages/booking/list.vue`)
- **按钮文案变更**：将所有 `签到` 按钮文案改为 `核销码`。
- **点击交互**：点击 `核销码` 按钮，不再弹出“确认签到”对话框，而是打开**核销码展示弹窗**。

### 3.2 详情页 (`src/pages/booking/detail.vue`)
- **按钮文案变更**：同上，`签到` -> `核销码`。
- **状态描述优化**：`PAID` 状态描述改为“请向前台出示核销码”。
- **新增组件/弹窗**：`VerifyCodeModal`（核销码弹窗）。
  - **内容**：
    - 标题：“请向前台出示此码”
    - 核心内容：**大号加粗的订单号**（分段显示更佳，如 `2023 0520 1234`）。
    - 提示文案：“凭此码核销入场”。

### 3.3 状态管理 (`src/stores/booking.js`)
- 无需变更核心逻辑，仅需确保 `orderNo` 字段在列表和详情接口中都能正确返回（目前已支持）。

## 4. 执行步骤
1.  **修改 `list.vue`**：将“签到”按钮改为“核销码”，点击事件改为 `showVerifyCodeModal(booking)`。
2.  **修改 `detail.vue`**：将“签到”按钮改为“核销码”，点击事件改为 `showVerifyCodeModal`。
3.  **实现 `VerifyCodeModal` 逻辑**：
    - 在 `list.vue` 和 `detail.vue` 中引入/实现一个简单的弹窗，展示 `orderNo`。
    - 样式上突出显示订单号。

## 5. 预期效果
用户支付后 -> 点击“核销码” -> 手机展示大号订单号 -> 管理员在后台/手持端输入订单号 -> 核销成功 -> 订单状态变更为 `VERIFIED`（已核销）。
