# 球场管理员端后端开发指南 (Backend Development Guide)

## 1. 文档概述
本文档基于《球场管理员端产品需求文档 (PRD)》及现有后端代码现状（`api_documentation.md`）编写。旨在指导后端开发者完成管理员端功能的落地。

- **核心原则**：
  - **复用优先**：尽量复用现有的 Controller 和 Service 逻辑。
  - **权限隔离**：严格校验 `ROLE_VENUE_ADMIN` 权限。
  - **最小改动**：在不破坏用户端功能的前提下进行扩展。

### 1.1 当前实现进度（代码已落地）
- ✅ 已实现 `GET /api/admin/dashboard/stats`（管理员工作台统计）
- ✅ 已实现 `GET /api/venues/manager/me`（当前管理员管理场馆）
- ✅ 已实现 `GET /api/verification/code/{code}` + `POST /api/verification/code/verify`（ORD/REQ 核销码统一查询与核销）
- ✅ 已实现 `GET /api/admin/bookings`（管理员订单列表筛选）
- ✅ 已实现 `POST /api/bookings/{id}/admin-cancel`（管理员取消，拼场级联取消与逻辑退款）
- ✅ 已实现 `PATCH /api/timeslots/{id}/status` 冲突校验（有订单时禁止设为维护）
- ✅ 已实现拼场详情 participants 返回（`GET /api/bookings/{id}`）
- ✅ 已实现 `Venue.autoGenerateDays` 字段与每日滚动补齐定时任务
- ✅ 已补充场馆配置兼容字段：`coverImage -> image`、`facilityTags -> features`、`contactPhone`

## 2. 鉴权与账号体系 (Auth & Account)

### 2.1 现状分析
- **现有能力**：`AuthController` 已支持登录 (`/api/auth/signin`)，JWT 包含角色信息。
- **PRD 要求**：管理员账号由运营方**手动在数据库创建**，不支持自助注册。
- **差异**：现有 `signup` 接口默认创建 `ROLE_USER`，符合不开放管理员注册的需求。

### 2.2 开发任务
1.  **账号创建（运维侧）**：
    - 无需开发接口。直接通过 SQL 语句在 `users` 表插入管理员账号，并在 `user_roles` 表关联 `ROLE_VENUE_ADMIN`。
    - *SQL 示例*：
      ```sql
      INSERT INTO users (username, password, nickname, email, phone, active)
      VALUES ('admin01', '$2a$10$EncryptedPassword...', '场馆管理员A', 'admin@gym.com', '13800138000', 1);
      INSERT INTO user_roles (user_id, role)
      VALUES ((SELECT id FROM users WHERE username='admin01'), 'ROLE_VENUE_ADMIN');
      ```

2.  **登录接口（复用）**：
    - **接口**：`POST /api/auth/signin`
    - **逻辑**：保持不变。前端登录成功后，解析 JWT 中的 `roles` 字段，若包含 `ROLE_VENUE_ADMIN` 则跳转管理员端。

3.  **修改密码（复用）**：
    - **接口**：`PUT /api/users/me/password`
    - **逻辑**：保持不变。管理员登录后调用此接口修改自己的密码。

## 3. 工作台 (Dashboard) - **[新增模块]**

### 3.1 现状分析
- **现有能力**：无管理员专属数据统计接口。`VerificationController` 有核销逻辑但缺乏聚合数据。
- **PRD 要求**：数据概览（今日/本周/本月订单数、收益等）、快捷核销。

### 3.2 开发任务
1.  **新增统计接口**：
    - **接口**：`GET /api/admin/dashboard/stats`
    - **参数**：`timeRange` (today/week/month/custom), `startDate`, `endDate`
    - **逻辑**：
        - 获取当前管理员管理的场馆 ID 列表。
        - 查询 `bookings` 表，统计指定时间范围内、指定场馆的订单数据。
        - **待核销数**：状态为 `PAID` 或 `SHARING_SUCCESS`。
        - **已核销数**：状态为 `VERIFIED` 或 `COMPLETED`。
        - **收益**：状态为 `COMPLETED` 的订单总金额。
    - **Controller**：建议新建 `AdminDashboardController` 或在 `VenueController` 中添加。

2.  **快捷核销查询（新增）**：
    - **接口**：`GET /api/verification/code/{code}`
    - **逻辑**：根据核销码查询订单详情，用于管理员扫码或输码后的回显。
    - **复用点**：可参考 `VerificationController` 的逻辑，但需返回完整的订单展示信息（含拼场信息）。

## 4. 订单管理 (Order Management) - **[需增强]**

### 4.1 现状分析
- **现有能力**：`BookingController.getBookings` 支持按状态、日期筛选。
- **PRD 要求**：支持按**手机号**、**核销码**搜索；支持**退款**。
- **拼场逻辑**：用户端存在主订单（Booking）和虚拟订单（SharingRequest）。管理员端**列表仅展示主订单**，但在详情中需展示所有参与者。

### 4.2 开发任务
1.  **增强列表查询接口**：
    - **接口**：`GET /api/admin/bookings` (或复用 `GET /api/bookings` 并增强)
    - **新增参数**：`keyword` (匹配手机号或核销码)、`venueId` (筛选特定场馆)、`type` (普通/拼场)。
    - **逻辑**：
        - 在 `BookingRepository` 中增加支持模糊搜索的查询方法。
        - **拼场特殊处理**：
            - 若 keyword 匹配到 `REQ_` 开头的核销码（参与者），需反查出对应的**主订单**返回。
            - 列表结果中**排除**虚拟订单，只返回主订单。
        - 确保管理员只能看到自己管理场馆的订单。

2.  **订单详情增强（拼场支持）**：
    - **接口**：`GET /api/bookings/{id}`
    - **逻辑**：
        - 若订单类型为 `SHARED`，需额外查询 `SharingOrder` 及关联的 `SharingRequest` 列表。
        - 返回结构中增加 `participants` 字段，包含所有参与者信息（昵称、手机号、核销状态）。

3.  **实现退款/取消接口（新增）**：
    - **接口**：`POST /api/bookings/{id}/admin-cancel`
    - **逻辑**：
        - 仅限 `PAID` 或 `SHARING_SUCCESS` 状态订单。
        - **拼场级联取消**：若为主订单，需同时取消关联的 `SharingOrder` 及所有 `SharingRequest`。
        - **批量退款**：遍历所有已支付的参与者（包括发起者），调用退款逻辑。
        - 释放对应的 `TimeSlot` 资源（设为 `AVAILABLE`）。
        - **注意**：需区别于用户端取消（用户端可能有违约金规则），管理员取消通常是全额退款。

## 5. 场地管理 (Venue Management) - **[需完善]**

### 5.1 现状分析
- **现有能力**：`VenueController` 支持增删改查；`TimeSlotController` 支持生成和状态更新。
- **PRD 要求**：上下架、拼场开关、**自动生成策略**、状态干预（锁场）。

### 5.2 开发任务
1.  **场馆列表（管理员视角）**：
    - **接口**：`GET /api/venues/manager/me`
    - **逻辑**：返回当前登录管理员负责的所有场馆。

2.  **更新场馆配置（复用/增强）**：
    - **接口**：`PUT /api/venues/{id}`
    - **增强**：
        - 确保前端传递 `supportSharing` (拼场开关) 字段时后端能正确保存。
        - 新增字段 `autoGenerateDays` (自动生成未来N天)，需修改 `Venue` 实体和数据库表。
        - 场馆配置字段按 PRD 对齐为：`name`、`type`、`coverImage`、`location`、`contactPhone`、`facilityTags`、`description`、`openTime`、`closeTime`、`price`、`supportSharing`、`autoGenerateDays`。

3.  **排期管理 - 状态干预（复用/增强）**：
    - **接口**：`PATCH /api/timeslots/{id}/status`
    - **逻辑**：
        - 现有逻辑支持更新状态。
        - **增强校验**：当管理员尝试将时段设为 `MAINTENANCE` (维护) 时，必须检查该时段是否已有有效订单（`PAID` 等）。若有，抛出异常提示“存在冲突订单，请先处理”。

4.  **时间段自动生成任务（核心逻辑）**：
    - **逻辑**：
        - 目前 `TimeSlotService` 可能仅支持按天生成。
        - 需创建一个定时任务 (`@Scheduled`)，每天凌晨运行。
        - 遍历所有场馆，读取其 `autoGenerateDays` 配置（如7天）。
        - 检查未来7天内的 `TimeSlot` 是否存在，不存在则调用 `generateTimeSlotsForVenue` 补齐。

## 6. 核销模块 (Verification) - **[复用]**

### 6.1 现状分析
- **现有能力**：`VerificationController` 已实现 `verify` 和 `complete`。
- **差异点**：现有接口仅支持 `Long` 类型 ID，无法处理拼场参与者的 `REQ_xxx` 格式核销码。
- **PRD 要求**：输入核销码 -> 验证 -> 确认。

### 6.2 开发任务
1.  **核销码解析与路由（核心改造）**：
    - **接口**：`POST /api/verification/code/verify` (建议新增通用入口，替代 `/orders/{id}/verify`)
    - **参数**：`code` (String, 支持 `ORD...` 或 `REQ_...`)
    - **逻辑**：
        - 若 code 以 `ORD` 开头：按 `orderNo` 查询 `bookings` 表，获取主订单。
        - 若 code 以 `REQ_` 开头：提取 ID，查询 `sharing_requests` 表，获取关联的主订单。
        - **状态更新**：
            - 记录当前核销的具体人员（发起者或参与者）。
            - 更新主订单状态为 `VERIFIED`（策略：任一有效核销即视为该场次开始使用）。

2.  **核销确认接口（兼容旧版）**：
    - **接口**：`POST /api/verification/orders/{id}/verify`
    - **逻辑**：保持不变，仅用于已知主订单 ID 的场景。

## 7. 数据库变更建议 (Database Changes)

为了支持上述功能，建议执行以下数据库变更：

```sql
-- 1. 场馆表增加自动生成天数配置
ALTER TABLE venues ADD COLUMN auto_generate_days INT DEFAULT 7 COMMENT '自动生成未来多少天的排期';

-- 2. 场馆表增加联系电话（用于场馆配置字段 contactPhone）
ALTER TABLE venues ADD COLUMN contact_phone VARCHAR(30) NULL COMMENT '场馆联系电话';

-- 3. 索引检查（order.order_no 与 users.phone 通常已由唯一约束提供索引）
SHOW INDEX FROM `order` WHERE Column_name = 'order_no';
SHOW INDEX FROM users WHERE Column_name = 'phone';
```

### 7.1 你手动执行数据库变更的步骤（推荐）
1. 先执行 `ALTER TABLE venues ... auto_generate_days`。
2. 再执行 `ALTER TABLE venues ... contact_phone`。
3. 执行后用以下 SQL 自检：
```sql
SHOW COLUMNS FROM venues LIKE 'auto_generate_days';
SHOW COLUMNS FROM venues LIKE 'contact_phone';
SHOW INDEX FROM `order` WHERE Column_name = 'order_no';
SHOW INDEX FROM users WHERE Column_name = 'phone';
```
4. 若索引已存在无需重复创建，避免重复建索引报错。

## 9. 技术实现细节 (Technical Implementation Details)

为了确保代码质量和一致性，开发时请遵循以下规范：

1.  **权限控制**：
    - 所有管理员接口必须添加注解：`@PreAuthorize("hasRole('VENUE_ADMIN')")`。
    - 在 Service 层需二次校验数据权限：确保当前管理员 ID (`UserUtils.getCurrentUserId()`) 与请求操作的场馆的 `managerId` 一致。

2.  **数据传输对象 (DTO)**：
    - **禁止**直接在 Controller 返回 Entity 实体。
    - 需新建 DTO 类，例如：
        - `AdminStatsResponse`: 用于仪表盘数据。
        - `AdminOrderDTO`: 用于订单列表，包含 `userPhone` (来自 User 表) 和 `participants` (拼场成员列表)。

3.  **复杂查询**：
    - 订单列表的搜索（关键词+状态+场馆+时间）建议使用 `JpaSpecificationExecutor` 实现动态 SQL 拼接，避免手写复杂 `@Query`。

4.  **时间处理**：
    - 统计“今日”数据时，请使用系统默认时区（或配置时区）的 `00:00:00` 到 `23:59:59`。

5.  **异常处理**：
    - 遇到业务错误（如“核销码无效”、“无权操作”），请抛出 `BusinessException` 或 `ResponseStatusException`，由全局异常处理器捕获。

## 12. 复杂查询入参格式与组合示例

### 12.1 管理员订单列表 `GET /api/admin/bookings`
- **Query 参数定义**
  - `page`：页码，起始 1
  - `pageSize`：每页条数
  - `status`：订单状态（如 `PAID`、`VERIFIED`、`COMPLETED`、`CANCELLED`）
  - `keyword`：手机号片段 / `ORD...` / `REQ_xxx`
  - `venueId`：场馆 ID
  - `type`：`EXCLUSIVE`（普通） / `SHARED`（拼场）
  - `startDate`：`yyyy-MM-dd`
  - `endDate`：`yyyy-MM-dd`

- **组合示例 A：待核销拼场单**
  - `/api/admin/bookings?page=1&pageSize=20&status=PAID&type=SHARED`

- **组合示例 B：按参与者核销码反查主单**
  - `/api/admin/bookings?page=1&pageSize=10&keyword=REQ_88`

- **组合示例 C：按手机号+日期范围查询**
  - `/api/admin/bookings?page=1&pageSize=10&keyword=1380013&startDate=2026-03-01&endDate=2026-03-31`

### 12.2 工作台统计 `GET /api/admin/dashboard/stats`
- **timeRange 固定值**
  - `today`：今日
  - `week`：本周（周一到周日）
  - `month`：本月
  - `custom`：自定义范围（必须同时传 `startDate` 与 `endDate`）

- **组合示例**
  - 今日：`/api/admin/dashboard/stats?timeRange=today`
  - 自定义：`/api/admin/dashboard/stats?timeRange=custom&startDate=2026-03-01&endDate=2026-03-12`

### 12.3 场馆列表 `GET /api/venues/manager/me`
- **组合示例**
  - `/api/venues/manager/me`

## 13. 错误码与异常返回示例（联调标准）

### 13.1 建议统一错误体
```json
{
  "success": false,
  "code": "FORBIDDEN",
  "message": "无权限操作该资源",
  "path": "/api/timeslots/100/status",
  "timestamp": "2026-03-12T23:30:00"
}
```

### 13.2 当前已落地接口中的常见错误码
- `400 Bad Request`
  - 参数非法、状态流转不允许、核销码格式错误
- `401 Unauthorized`
  - 未登录或令牌失效
- `403 Forbidden`
  - 角色不匹配或非本人管理场馆
- `404 Not Found`
  - 订单/用户/场馆/时段不存在
- `409 Conflict`
  - 锁场冲突（时段已有订单，不允许设为维护）
- `500 Internal Server Error`
  - 服务内部异常（如刷新时段失败）

### 13.3 典型异常示例
1. **核销码不存在（400）**
```json
{
  "message": "核销码不存在"
}
```

2. **时段锁场冲突（409）**
```json
{
  "success": false,
  "message": "存在冲突订单，无法设为维护中，请先处理订单",
  "timeSlotId": 1234
}
```

3. **越权取消订单（403）**
```json
{
  "success": false,
  "message": "无权限取消该订单"
}
```

4. **统计参数错误（400）**
```json
{
  "success": false,
  "message": "自定义时间范围必须提供 startDate 和 endDate"
}
```

### 13.4 前端联调容错建议
- 前端先判断 `HTTP Status`，再读 `message` 字段做提示。
- 对 401 统一跳登录；对 403 显示“无权限”；对 409 显示“存在冲突订单，请先处理”。
- 对空 body 的 404/400 做兜底提示：`请求失败，请稍后重试`（兼容历史接口）。
