# 体育馆预约系统技术说明文档

## 1. 系统名称

基于Uniapp+SpringBoot的体育馆预约微信小程序设计与实现

## 2. 系统目标

本系统旨在为用户提供便捷的体育馆场地预约服务，支持个人包场预约与拼场（共享）预约模式，同时为场馆管理员提供高效的场地管理、订单处理及核销验证功能。系统基于微信小程序生态，通过前后端分离架构实现高性能、高可用的业务处理。

## 3. 技术栈

### 3.1 前端 (用户端/管理端)

- **框架**: uni-app (基于 Vue 3)
- **构建工具**: Vite
- **状态管理**: Pinia
- **UI 组件**: 自定义组件 + uni-ui
- **运行环境**: 微信小程序 (MP-WEIXIN)
- **开发工具**: HBuilderX

### 3.2 后端 (服务端)

- **核心框架**: Spring Boot 3.4.5
- **安全框架**: Spring Security + JWT (JSON Web Token)
- **持久层**: Spring Data JPA + Hibernate
- **数据库**: MySQL 8.0
- **构建工具**: Maven
- **API 文档**: SpringDoc OpenAPI (Swagger UI)
- **工具库**: Lombok, Commons-Lang3

### 3.3 开发与部署

- **JDK 版本**: Java 17
- **代码管理**: Git

## 4. 系统角色

系统设计了基于角色的访问控制 (RBAC)，主要包含以下角色：

1. **ROLE\_USER (普通用户)**:
   - 通过微信一键登录。
   - 拥有浏览场馆、预约场地、发起/加入拼场、查看个人订单、支付等权限。
2. **ROLE\_VENUE\_ADMIN (场馆管理员)**:
   - 负责管理下场馆的维护、订单管理及现场核销。

## 5. 系统模块划分

系统主要分为以下核心业务模块：

1. **认证模块 (Auth Module)**: 处理用户注册、登录（账号密码/微信授权）、JWT 令牌发放与校验。
2. **用户模块 (User Module)**: 管理用户个人信息、头像、昵称及角色分配。
3. **场馆模块 (Venue Module)**: 场馆信息的增删改查、状态管理（开放/维护）、价格设置。
4. **预约模块 (Booking Module)**: 时间段（TimeSlot）管理、普通订单创建、支付、取消、过期处理。
5. **拼场模块 (Sharing Module)**: 拼场订单的发起、申请加入、审核、拼场状态流转。
6. **支付模块 (Payment Module)**: 支付流程、支付状态回调及记录。
7. **管理模块 (Admin Module)**: 仪表盘数据统计、核销验证、安全设置。

## 6. 用户端功能

- **首页**: 展示推荐场馆、热门活动及快捷入口。
- **场馆列表**: 支持按类型、状态筛选场馆，查看场馆详情（设施、图片、位置）。
- **场地预约**:
  - 查看场馆排期（时间段状态：可预约、已预约、维护中）。
  - 选择时间段进行包场预约。
  - 在线支付（模拟）。
- **拼场专区**:
  - 发布拼场需求（招募球友）。
  - 查看正在进行的拼场。
  - 申请加入拼场队伍。
- **个人中心**:
  - 查看我的订单（全部、待支付、待使用、已完成、已取消）。
  - 查看我的拼场（发起的、参与的）。
  - 个人信息修改。
- **订单核销**: 展示订单核销码（订单号），供管理员现场验证。

## 7. 管理员端功能

- **工作台 (Dashboard)**:
  - 实时数据概览：今日订单数、待核销数、今日收入。
  - 快捷入口：扫码核销、场地管理、订单查询。
- **订单管理**:
  - 查看所有订单列表。
  - 订单筛选与详情查看。
  - 手动取消违规或异常订单。
- **场地管理**:
  - 新增/编辑场馆信息。
  - 设置场馆开放时间与价格。
  - 发布/维护时间段状态。
- **核销验证**:
  - 输入订单号进行入场核销。
  - 查看核销历史。
- **安全中心**: 修改管理员密码。

## 8. 系统架构

采用经典的 **Client-Server** 分层架构：

1. **Client Layer (前端)**:
   - uni-app 构建的微信小程序，负责 UI 渲染、用户交互及本地业务逻辑。
   - 通过 HTTPS 请求与后端通信，使用 Pinia 进行全局状态管理。
2. **API Layer (接口层)**:
   - Spring Boot Controller 暴露 RESTful API。
   - Spring Security 过滤器链处理跨域 (CORS) 与 身份认证 (JWT Filter)。
3. **Service Layer (业务层)**:
   - 封装核心业务逻辑（如订单状态机、拼场匹配算法）。
   - 事务管理 (@Transactional)。
4. **Data Layer (数据层)**:
   - JPA Repository 接口与 Hibernate 实现数据持久化。
   - MySQL 数据库存储关系型数据。

## 9. 前后端交互流程

1. **请求发起**: 前端封装 `request.js` 工具类，统一处理 URL 前缀、超时设置。
2. **鉴权注入**: 请求拦截器自动在 Header 中携带 `Authorization: Bearer <token>`。
3. **请求处理**: 后端 Controller 接收请求，Service 处理业务，Repository 操作数据库。
4. **响应返回**: 后端统一返回 JSON 格式数据。
5. **异常处理**:
   - 后端：GlobalExceptionHandler 捕获异常并返回标准错误结构。
   - 前端：响应拦截器统一处理 401（未登录/过期）、403（无权限）、500（服务器错误）等状态码，并进行 Toast 提示或跳转登录。

## 10. 数据库表结构

核心数据表如下：

1. **users (用户表)**:
   - `id`: 主键
   - `username`, `password`: 账号密码
   - `wechat_openid`: 微信唯一标识
   - `role`: 用户角色关联
2. **venues (场馆表)**:
   - `id`: 主键
   - `name`: 场馆名称
   - `price`: 基础价格
   - `status`: 状态 (OPEN, CLOSED, MAINTENANCE)
   - `support_sharing`: 是否支持拼场
3. **time\_slots (时间段表)**:
   - `venue_id`: 关联场馆
   - `date`, `start_time`, `end_time`: 时间信息
   - `status`: 状态 (AVAILABLE, RESERVED, OCCUPIED, etc.)
   - `order_id`: 关联订单（若被预订）
   - **唯一约束**: `venue_id` + `date` + `start_time` + `end_time`
4. **order (订单表)**:
   - `id`: 主键
   - `order_no`: 唯一订单号
   - `user_id` (逻辑关联 username): 下单用户
   - `status`: 订单状态 (PENDING, PAID, VERIFIED, CANCELLED, etc.)
   - `booking_type`: 预约类型 (EXCLUSIVE-包场, SHARING-拼场)
5. **sharing\_orders (拼场主单表)**:
   - `id`: 主键
   - `venue_id`: 关联场馆
   - `status`: 拼场状态 (OPEN, FULL, COMPLETED, etc.)
   - `current_participants`, `max_participants`: 人数控制
6. **sharing\_request (拼场申请表)**:
   - `sharing_order_id`: 关联拼场单
   - `applicant_username`: 申请人
   - `status`: 申请状态 (PENDING, APPROVED, REJECTED)

## 11. 核心接口

### 认证接口 (`/api/auth`)

- `POST /signin`: 用户/管理员登录
- `POST /wechat/login`: 微信小程序登录

### 场馆接口 (`/api/venues`)

- `GET /`: 获取场馆列表（支持分页、筛选）
- `POST /`: 创建场馆（管理员）
- `PUT /{id}`: 更新场馆信息

### 预约接口 (`/api/order` & `/api/timeslots`)

- `POST /api/order`: 创建订单
- `GET /api/order/all`: 获取所有订单（管理员）
- `PUT /api/order/{id}/cancel`: 取消订单
- `GET /api/timeslots`: 获取时间段状态（可用/已占用）

### 拼场接口 (`/api/shared`)

- `POST /`: 发起拼场
- `POST /{id}/join`: 申请加入
- `PUT /requests/{requestId}`: 处理拼场申请

### 验证接口 (`/api/verification`)

- `POST /verify`: 核销订单

## 12. 系统运行流程

### 12.1 普通预约流程

1. 用户登录 -> 进入场馆列表 -> 选择目标场馆。
2. 查看场馆排期 -> 选择状态为“可用”的时间段。
3. 确认订单信息 -> 提交订单（生成 PENDING 状态订单，锁定时间段）。
4. 模拟支付 -> 订单变更为 PAID 状态 -> 时间段变更为 BOOKED。
5. 用户到场 -> 出示订单号。
6. 管理员核销 -> 订单变更为 VERIFIED -> 流程结束。

### 12.2 拼场流程

1. 用户A（发起者）选择支持拼场的场馆 -> 选择时间段 -> 选择“拼场模式”。
2. 填写拼场信息（队伍名称、招募人数、费用分摊） -> 支付个人份额 -> 生成拼场单。
3. 用户B（参与者）在拼场列表看到该拼场 -> 发起加入申请。
4. 用户A 收到通知 -> 审核通过。
5. 用户B 支付份额 -> 加入成功。
6. 拼场人数达到上限或时间截止 -> 拼场锁定。
