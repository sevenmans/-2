# 数据库表结构分析与详情报告

## 1. 核心表名列表

系统核心业务数据存储在以下 6 张物理表中：

1.  **`users`**: 用户信息表
2.  **`venues`**: 场馆信息表
3.  **`time_slots`**: 场地排期时间段表
4.  **`order`**: 订单基础信息表
5.  **`sharing_orders`**: 拼场主订单表
6.  **`sharing_request`**: 拼场申请记录表

---

## 2. 表之间关系梳理 (ER 关系)

系统的表关系设计以 **逻辑外键** 为主，避免了物理外键的强约束，提升了系统的灵活性：

1.  **用户与场馆 (User -> Venue)**
    *   **1 对 多**: 一名管理员 (`users.id`) 可以管理多个场馆 (`venues.manager_id`)。
2.  **场馆与时间段 (Venue -> TimeSlot)**
    *   **1 对 多**: 一个场馆 (`venues.id`) 拥有多个排期时间段 (`time_slots.venue_id`)。
3.  **用户与订单 (User -> Order)**
    *   **1 对 多**: 一个用户 (`users.username`) 可以创建多个订单 (`order.username`)。
4.  **订单与时间段 (Order <-> TimeSlot)**
    *   **1 对 1 (逻辑)**: 订单生成后，通过锁定 `time_slots` 记录（回写 `order_id`）实现占用。
5.  **订单与拼场 (Order -> SharingOrder)**
    *   **1 对 1**: 当预约类型为拼场时，一条基础订单 (`order.id`) 对应一条拼场详情记录 (`sharing_orders.order_id`)。
6.  **拼场单与申请记录 (SharingOrder -> SharingRequest)**
    *   **1 对 多**: 一个拼场招募 (`sharing_orders.id`) 可以接收多个用户的加入申请 (`sharing_request.sharing_order_id`)。

| 字段名 | 数据类型 | 长度 | 是否为空 | 默认值 | 说明 |

| `password` | VARCHAR | 120 | 否 | 无 | 加密密码 |
| `nickname` | VARCHAR | 50 | 否 | 无 | 用户昵称 |
| `phone` | VARCHAR | 20 | 是 | NULL | 手机号码(唯一) |
| `email` | VARCHAR | 50 | 是 | NULL | 电子邮箱(唯一) |
| `avatar` | VARCHAR | 255 | 是 | NULL | 头像 URL |
| `wechat_openid` | VARCHAR | 64 | 是 | NULL | 微信 OpenID(唯一) |
| `active` | BIT | 1 | 否 | 1 | 账号状态(1可用 0禁用) |
| `created_at` | DATETIME | - | 否 | CURRENT_TIMESTAMP | 创建时间 |
| `updated_at` | DATETIME | - | 否 | CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |
