# 系统非功能需求文档 (System Non-Functional Requirements)

## 1. 系统性能特点 (Performance)

### 1.1 响应速度优化
*   **前端智能缓存**: 实现了 `SmartCache` 和 `CacheManager` 多级缓存机制（内存 + LocalStorage），对 GET 请求进行智能缓存，有效减少网络请求频率，提升页面加载速度。
*   **请求优化**: 引入了 `requestDeduplicator`（请求去重）和 `concurrencyController`（并发控制），防止重复点击导致的资源浪费，确保高并发场景下的前端稳定性。
*   **资源加载**: 采用 uni-app 的分包加载机制，减小首屏包体积，提升小程序启动速度。

### 1.2 后端并发处理
*   **数据库连接池**: 使用 HikariCP 高性能连接池管理 MySQL 连接，确保在高并发请求下的数据库访问效率。
*   **异步处理**: 引入 Spring `@Async` 和 `Scheduled` 任务调度，将订单过期检查、拼场超时处理等耗时操作异步化，避免阻塞主业务线程。
*   **无状态架构**: 基于 JWT 的无状态认证机制，使得后端服务可以轻松进行水平扩展（Scale-out），不受 Session 共享限制。

## 2. 系统安全机制 (Security)

### 2.1 认证与授权
*   **JWT 令牌认证**: 采用 `Spring Security` + `JWT` 标准认证流程。用户登录后获取 Access Token，后续请求需在 Header 中携带 `Bearer Token`。
*   **RBAC 权限控制**: 基于角色的访问控制（Role-Based Access Control），严格区分 `ROLE_USER`、`ROLE_VENUE_ADMIN` 和 `ROLE_SUPER_ADMIN` 的权限边界。例如，核销接口 `@PreAuthorize("hasRole('ROLE_VENUE_ADMIN')")` 确保只有管理员可调用。
*   **无状态会话**: 禁用 HttpSession (`SessionCreationPolicy.STATELESS`)，防止 CSRF 攻击，并降低服务器内存开销。

### 2.2 数据安全
*   **密码加密**: 用户密码存储采用 `BCryptPasswordEncoder` 进行强哈希加密，防止数据库泄露后的彩虹表攻击。
*   **HTTPS 通信**: 生产环境强制开启 SSL/HTTPS 加密传输，确保用户数据在传输过程中的安全性（开发环境可配置关闭）。
*   **参数校验**: 后端使用 `@Valid` 和 `@NotNull` 等注解对入参进行严格校验，防止 SQL 注入和非法数据录入。

### 2.3 业务安全
*   **防重复提交**: 前端实现了请求去重拦截器，后端关键接口（如订单创建、支付）具备幂等性设计，防止用户重复扣款或重复下单。
*   **核销码机制**: 订单生成唯一的核销码，配合管理员现场验证，确保入场凭证的不可伪造性。

## 3. 系统可维护性 (Maintainability)

### 3.1 代码规范与架构
*   **分层架构**: 严格遵循 `Controller` -> `Service` -> `Repository` -> `Model` 的分层模式，职责清晰，降低耦合度。
*   **统一异常处理**: 实现了全局异常处理器 (`GlobalExceptionHandler`)，统一捕获并规范化返回错误信息，便于前端统一处理和日志排查。
*   **RESTful API**: 接口设计遵循 RESTful 规范，语义清晰（GET/POST/PUT/DELETE），易于理解和维护。

### 3.2 日志与监控
*   **结构化日志**: 使用 Logback 记录详细的业务日志，包含时间戳、线程ID、日志级别和上下文信息，便于故障定位。
*   **健康检查**: 集成 Spring Boot Actuator，提供 `/actuator/health` 和 `/actuator/metrics` 端点，实时监控应用健康状态和性能指标。
*   **定时任务监控**: `OrderStatusScheduler` 定期检查订单状态一致性，自动修复超时未支付或异常状态的订单，减少人工运维成本。

## 4. 系统可扩展性 (Scalability)

### 4.1 模块化设计
*   **业务模块解耦**: 预约、拼场、支付、用户管理等核心模块在逻辑上高度解耦。例如，拼场模块 (`SharingService`) 独立于基础订单模块，未来可独立拆分为微服务。
*   **接口版本控制**: API 设计预留了版本扩展能力，便于后续迭代升级而不影响旧版小程序客户端。

### 4.2 数据库扩展
*   **JPA/Hibernate**: 使用 Spring Data JPA 屏蔽了底层 SQL 差异，支持通过配置切换不同数据库（如从 MySQL 迁移至 PostgreSQL）。
*   **读写分离准备**: 数据库连接配置支持主从复制模式，未来可轻松升级为读写分离架构以应对更高流量。

### 4.3 缓存扩展
*   **多级缓存接口**: 前端封装了统一的 `CacheManager` 接口，目前使用 LocalStorage，未来可无缝切换至 IndexedDB 或其他存储引擎。
*   **Redis 预留**: 后端架构支持引入 Redis 作为分布式缓存（如存储 Token 黑名单、热点场馆数据），提升系统吞吐量。
